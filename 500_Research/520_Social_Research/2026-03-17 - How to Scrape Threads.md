# Как парсить Threads

Краткий вывод: **парсить Threads можно, но это неприятная цель**. В 2026 по обсуждениям и репозиториям видно 3 основных пути:

1. **Официальный Threads API** — самый чистый путь, но ограниченный.
2. **Reverse engineering внутренних GraphQL/Instagram endpoints** — работает точечно, но хрупко.
3. **Headless browser / screen scraping (Playwright)** — самый практичный для публичных страниц, но самый тяжёлый и нестабильный в долгую.

---

## 1. Что говорят форумы и репозитории

### Reddit / webscraping
По найденным обсуждениям в r/webscraping:
- ранние обсуждения Threads сводились к тому, что веб-версия ограничена и многое приходится тянуть через внутренние JSON/GraphQL ответы;
- для публичных комментариев и reply-chain люди искали JSON с `thread_items`;
- позже обсуждения сместились в сторону аккаунтов, headless-браузеров и обхода ограничений.

Вывод: сообщество не нашло одного «красивого» способа. Все крутятся вокруг вытаскивания уже отрендеренных данных или внутренних запросов.

### GitHub / reverse engineering
По `m1guelpf/threads-re`:
- веб-версия использует **Meta Relay GraphQL Client**;
- backend общается через `https://www.threads.net/api/graphql`;
- произвольные GraphQL query, судя по описанию, не приветствуются/не работают;
- используются фиксированные `doc_id` и заранее известные frontend-запросы;
- для вызовов нужен хотя бы корректный `user-agent` и `x-ig-app-id: 238260118697367`;
- автор также описывал auth через Instagram/Bloks и прямо помечал это как хрупкий reverse engineering.

Вывод: **внутренний GraphQL есть**, но это не полноценный открытый API, а набор внутренних контрактов, которые Meta может менять без предупреждения.

### GitHub / production scraper-подход
По `Zeeshanahmad4/Threads-Scraper`:
- ставка сделана не на API, а на **browser automation**;
- используется Playwright + парсинг уже отрендеренной страницы;
- заявлена работа без логина на публичных страницах;
- есть rate limiting, retry, scrolling, structured output.

Вывод: на практике рынок пришёл к тому, что **если нужны публичные данные, проще часто не ломать private API, а симулировать пользователя в браузере**.

### Meta docs / официальная позиция
По Meta Threads docs и developer discussions:
- есть **официальный Threads API** через `graph.threads.net`;
- он подходит для легитимной работы со своим приложением/аккаунтом;
- но не даёт тебе нормального «общего парсинга всего Threads», как это хочется исследователю/агрегатору;
- scraping у Meta в целом запрещён правилами, а официальный совет — использовать API там, где он покрывает кейс.

## 2. Реально рабочие варианты

## Вариант A — Официальный Threads API

### Когда подходит
- ты работаешь **со своим аккаунтом**;
- нужны публикации, аналитика, чтение своих сущностей, автоматизация контента;
- важна стабильность и легальность.

### Когда не подходит
- ты хочешь массово читать чужие профили/ленты/реплаи;
- тебе нужен конкурентный мониторинг;
- нужен full scraping публичного контента.

### Плюсы
- стабильнее;
- не надо ломать фронтенд;
- меньше риск блокировок.

### Минусы
- мало данных для исследовательского/мониторингового use case;
- жёстко ограничен продуктовой моделью Meta.

---

## Вариант B — Internal GraphQL / reverse engineering

### Как это обычно делают
1. Открывают Threads в браузере.
2. Смотрят Network.
3. Ловят POST на `threads.net/api/graphql`.
4. Вынимают `doc_id`, `variables`, headers.
5. Повторяют этот запрос скриптом.

### Что обычно нужно
- `user-agent`
- `x-ig-app-id`
- иногда куки / сессия
- корректный набор `doc_id` для нужных сущностей

### Где ломается
- Meta меняет `doc_id` и внутренние схемы;
- часть запросов требует auth/контекст;
- часть сценариев завязана на Instagram-side endpoints;
- с 2FA / логином всё быстро становится больно.

### Для чего годится
- точечный ресерч;
- прототип;
- разовый extractor;
- поиск структуры данных.

### Для чего плохо годится
- надёжный production pipeline на месяцы;
- продукт, который должен «просто работать».

---

## Вариант C — Playwright / browser automation

Это сейчас самый практичный вариант, если нужен **публичный контент**.

### Как выглядит схема
1. Playwright открывает профиль или пост.
2. Ждёт клиентский рендер.
3. Скроллит/раскрывает ветки.
4. Забирает DOM, текст, timestamp, счётчики, ссылки.
5. Складывает в JSON.

### Плюсы
- меньше завязки на private API-контракты;
- можно парсить то, что реально видит пользователь;
- хорошо подходит для публичных профилей/постов.

### Минусы
- медленно;
- жрёт CPU/RAM;
- сложнее масштабировать;
- UI может поменяться и селекторы отвалятся;
- Meta может ограничивать агрессивное поведение.

### Когда это лучший вариант
- нужен мониторинг нескольких аккаунтов;
- нужно вытаскивать посты/реплаи/метрики с публичных страниц;
- приемлема задержка и не нужен real-time firehose.

## 3. Практический вывод по use case

### Если цель — анализ своего профиля / автопостинг / метрики
Бери **официальный Threads API**.

### Если цель — разово исследовать, какие данные доступны у поста/профиля
Бери **Network + GraphQL reverse engineering**.

### Если цель — мониторить публичные аккаунты/посты/обсуждения
Бери **Playwright scraper**.

Именно так сейчас это выглядит наиболее реалистично.

## 4. Рекомендуемый стек, если делать самому

### Минимальный рабочий стек
- **Playwright** — навигация и рендеринг
- **Python / Node.js** — оркестрация
- **BeautifulSoup / Cheerio** — пост-обработка HTML
- **SQLite/Postgres** — хранение
- **Rate limiter + retry + backoff** — обязательно

### Что собирать
- profile URL
- post URL
- author handle
- post text
- timestamp
- likes / replies / reposts (если видны)
- thread/reply tree depth
- media links

### Что делать обязательно
- дедупликация по post ID / canonical URL
- логирование неудач
- fallback-селекторы
- ограничение скорости
- отдельный parser layer, чтобы можно было менять селекторы без переписывания движка

## 5. Главные риски

1. **Юридический / ToS риск**
   Meta scraping не любит. Для коммерческого продукта это уже не мелочь.

2. **Хрупкость private API**
   Всё, что завязано на `doc_id`, внутренние GraphQL-запросы, login flow — ломается без предупреждения.

3. **Хрупкость UI scraping**
   Изменили верстку/атрибуты/логику раскрытия reply — всё, привет.

4. **Антибот-ограничения**
   Агрессивные headless-пайплайны без pacing и прокси долго не живут.

## 6. Что бы я рекомендовал тебе

Если задача практическая, а не академическая, то так:

### Стратегия 1 — «чистая»
Использовать **официальный Threads API** для своего профиля и своей аналитики.

### Стратегия 2 — «рабочая»
Для конкурентного мониторинга / контент-анализа делать **небольшой Playwright scraper только публичных страниц**.

### Стратегия 3 — «исследовательская»
Параллельно держать маленький reverse-engineering notebook по GraphQL:
- сохранять удачные `doc_id`,
- логировать, что и как меняется,
- использовать это как ускоритель, но не как единственный фундамент.

То есть: **production = browser automation**,
**research = GraphQL reverse engineering**,
**официальная автоматизация = Threads API**.
