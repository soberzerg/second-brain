## Основные концепции

### Что такое субагент?

**Субагент** — это специализированный ИИ-агент, который создается и управляется родительским (orchestrator) агентом для выполнения изолированных задач.

### Зачем нужны субагенты?

1. **Context Preservation** — сохранение контекста для многошаговых задач
2. **Specialization** — специализация с domain-specific инструкциями
3. **Multi-Model Support** — использование разных моделей для разных задач
4. **Parallelization** — параллельное выполнение задач
5. **Context Bloat Prevention** — предотвращение переполнения контекста ("dumb zone")

### Ключевая статистика (2026)

- 📈 **1,445%** рост запросов на мультиагентные системы (Q1 2024 → Q2 2025)
- 🎯 **90.2%** улучшение производительности (Opus 4 + Sonnet 4 subagent vs solo Opus 4)
- 🏢 **40%** корпоративных приложений будут включать task-specific агентов к концу 2026
- ⚠️ **3-4 агента** — оптимальное количество для большинства кейсов

## Паттерны архитектуры

### 1. Orchestrator-Subagent Pattern (Доминирующий в 2026)

```
┌─────────────────────┐
│  Orchestrator Agent │  ← Главный агент (координатор)
│   (Lead Agent)      │
└──────────┬──────────┘
           │
           ├─────────────┬─────────────┬─────────────┐
           ▼             ▼             ▼             ▼
    ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
    │ Subagent │  │ Subagent │  │ Subagent │  │ Subagent │
    │    #1    │  │    #2    │  │    #3    │  │    #4    │
    │(Research)│  │  (Code)  │  │  (Test)  │  │ (Review) │
    └──────────┘  └──────────┘  └──────────┘  └──────────┘
```

**Характеристики:**
- Иерархическая модель
- Lead agent управляет всеми субагентами
- Простая координация
- Подходит для большинства use cases

### 2. Deep Agents Framework (LangChain)

**Два первоклассных примитива:**

1. **Subagents** — делегирование изолированным агентам
2. **Skills** — прогрессивное раскрытие возможностей

**Решаемая проблема:** Context bloat

```python
# Пример концепции (псевдокод)
orchestrator.delegate_to_subagent(
    task="Analyze codebase",
    context_isolation=True,
    model="claude-sonnet-4-5"
)
```

### 3. Connected Agents (Microsoft Azure AI Foundry)

```
Main Agent (Brain)
    │
    ├─→ Child Agent A (Specialist Domain 1)
    ├─→ Child Agent B (Specialist Domain 2)
    └─→ Child Agent C (Specialist Domain 3)
```

**Особенности:**
- Main agent как мозг
- Понимание → Разбиение → Делегирование
- Каждый child agent — узкий специалист

## Практические подходы

### Подход 1: Claude Code (Рекомендуется для AGIents)

#### Создание субагента

**Команда:**
```bash
/agents
```

**Шаги:**
1. Выбрать scope (project-level / user-level)
2. Сгенерировать или написать вручную
3. Настроить в Markdown файле с YAML frontmatter

**Пример конфига субагента:**
```yaml
---
name: code-reviewer
description: Специализированный агент для ревью кода
model: claude-sonnet-4-5
tools: [read, grep, edit]
system_prompt: |
  Ты эксперт по code review.
  Анализируй код на:
  - Security vulnerabilities
  - Code smells
  - Performance issues
  - Best practices violations
---
```

#### Per-Task Model Selection (ARM big.LITTLE)

| Модель | Тип задач | Аналог CPU Core |
|--------|-----------|----------------|
| **Opus** | Сложные рассуждения | Performance cores |
| **Sonnet** | Генерация кода | Balanced cores |
| **Haiku** | Валидация, быстрые ответы | Efficiency cores |

**Применение:**
```javascript
// Концептуальный пример
const agent = {
  orchestrator: 'opus-4-6',      // Сложная координация
  codeGenerator: 'sonnet-4-5',   // Основная работа
  validator: 'haiku-4-5',        // Быстрая проверка
  tester: 'sonnet-4-5'          // Запуск тестов
};
```

### Подход 2: Background Agents (Параллелизация)

**Концепция:** Фоновые агенты работают независимо от основного потока

```bash
# Запуск background agent
task --background --agent research \
  --prompt "Analyze competitors" \
  --output /tmp/research-results.md
```

**Применение:**
- Долгие исследования
- Параллельная обработка данных
- Мониторинг процессов

### Подход 3: Agent Swarms (Координация множества агентов)

**Для особо сложных задач:**
```
Task: Разработать полный микросервис

Orchestrator
    │
    ├─→ Architecture Agent   (Проектирование)
    ├─→ Backend Agent        (Go код)
    ├─→ Frontend Agent       (React код)
    ├─→ Testing Agent        (Тесты)
    ├─→ Security Agent       (Аудит)
    └─→ Documentation Agent  (Docs)
```

## Best Practices

### 1. Принцип минимализма

> **"Start with the simplest approach that works, and add complexity only when evidence supports it"**

- ✅ Начинай с 1 агента
- ✅ Добавляй субагентов только при необходимости
- ✅ Max 3-4 специализированных агента
- ❌ Не создавай >4 агентов без явной причины

### 2. Правильная декомпозиция задач

**Хорошие кандидаты для субагентов:**
- ✅ Исследование (research)
- ✅ Написание кода (coding)
- ✅ Тестирование (testing)
- ✅ Документирование (documentation)
- ✅ Валидация (validation)

**Плохие кандидаты:**
- ❌ Слишком узкие задачи (overhead > пользы)
- ❌ Задачи требующие много координации (bottleneck)

### 3. Context Management

**Проблема Context Bloat:**
- Когда контекст близок к лимиту → агент входит в "dumb zone"
- Качество решений падает

**Решение:**
```
Orchestrator (Main Context)
    │
    ├─→ Subagent 1 (Isolated Context A)
    ├─→ Subagent 2 (Isolated Context B)
    └─→ Subagent 3 (Isolated Context C)
```

### 4. Workflow с Completion Gates

**8-фазный workflow (по Gur Sannikov):**

```
Phase 1: Requirements Analysis
    ↓ [Gate: Requirements validated]
Phase 2: Architecture Design
    ↓ [Gate: ADR approved]
Phase 3: Implementation
    ↓ [Gate: Code generated]
Phase 4: Testing
    ↓ [Gate: Tests pass]
Phase 5: Security Audit
    ↓ [Gate: No vulnerabilities]
Phase 6: Documentation
    ↓ [Gate: Docs complete]
Phase 7: Code Review
    ↓ [Gate: Review passed]
Phase 8: Deployment Preparation
    ↓ [Gate: Ready for deploy]
```

**Gates** (контрольные точки):
- Автоматическая проверка критериев
- Git коммит при успешном прохождении
- Блокировка следующей фазы при провале

### 5. Error Handling

**Стратегии обработки ошибок:**

1. **Retry с экспоненциальным backoff**
   ```python
   max_retries = 3
   for attempt in range(max_retries):
       try:
           result = subagent.execute(task)
           break
       except ContextOverflowError:
           subagent.clear_context()
           retry()
   ```

2. **Fallback на другую модель**
   ```python
   try:
       result = opus_agent.complex_reasoning(task)
   except TimeoutError:
       result = sonnet_agent.simplified_reasoning(task)
   ```

3. **Escalation к человеку**
   ```python
   if subagent.stuck_attempts > 3:
       notify_human("Agent needs guidance on task X")
       pause_execution()
   ```

### 6. Monitoring & Observability

**Что отслеживать:**
- 📊 Token usage per subagent
- ⏱️ Execution time per task
- ✅ Success/failure rate
- 🔄 Number of retries
- 🧠 Context window utilization

**Инструменты:**
- LangSmith (для LangChain)
- Azure AI Studio (для MS stack)
- Custom logging (для самописных систем)

## Применение к AGIents.pro

### Концепция "Task-to-Subagent Mapping"

**Идея:** Платформа AGIents автоматически создает оптимальную структуру субагентов на основе задачи клиента.

#### Пример 1: ИИ-консультант для санатория (Amaks)

```
Main Agent: Customer Support Orchestrator
    │
    ├─→ FAQ Agent          (Быстрые ответы на типовые вопросы)
    ├─→ Booking Agent      (Проверка доступности, бронирование)
    ├─→ Pricing Agent      (Расчет стоимости, скидки)
    └─→ Personalization    (Рекомендации на основе профиля)
```

**Маппинг моделей:**
- FAQ Agent → Haiku (быстро и дешево)
- Booking Agent → Sonnet (баланс)
- Pricing Agent → Haiku (простые вычисления)
- Personalization → Opus (сложная логика)

#### Пример 2: Autonomous Coding Agent (Идея из Ideas/)

```
Main Agent: Project Manager
    │
    ├─→ Zettelkasten Agent     (Поиск в базе знаний)
    ├─→ Task Parser Agent       (Анализ задачи из таск-менеджера)
    ├─→ Code Generator Agent    (Написание кода)
    ├─→ Testing Agent           (Запуск тестов)
    ├─→ Code Review Agent       (Анализ качества - AI-код-ревью)
    └─→ PR Creator Agent        (Создание PR с описанием)
```

**Workflow:**
```
1. Task Parser → Забирает задачу из YouGile
2. Zettelkasten → Ищет релевантную архитектуру/паттерны
3. Code Generator → Пишет код с учетом контекста
4. Testing → Запускает тесты
5. Code Review → Проверяет качество (интеграция AI-код-ревью)
6. PR Creator → Создает PR, если все OK
7. Main Agent → Обновляет статус в YouGile
```

## Ключевые выводы

### ✅ DO

1. **Начинай просто** — 1 агент → добавляй субагентов по необходимости
2. **Max 3-4 субагента** для большинства задач
3. **Per-task model selection** — Haiku для простых задач, Opus для сложных
4. **Isolate context** — субагенты решают проблему context bloat
5. **Monitor metrics** — token usage, cost, performance
6. **Use completion gates** — контрольные точки для качества

### ❌ DON'T

1. **Не создавай >4 субагентов** без явной причины (overhead)
2. **Не используй мультиагентность** если задача решается одним агентом
3. **Не игнорируй cost optimization** — Opus везде = $$$
4. **Не забывай error handling** — субагенты могут зависать
5. **Не копируй сложные архитектуры** без понимания зачем

### 🎯 Золотое правило

> **"The right amount of complexity is the minimum needed for the current task"**

Три похожие задачи лучше решить одним агентом, чем создавать преждевременную абстракцию из субагентов.
