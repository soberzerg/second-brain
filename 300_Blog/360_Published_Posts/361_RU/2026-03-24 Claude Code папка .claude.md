# Полный гид по папке .claude: CLAUDE.md, команды, навыки, агенты и permissions

> Адаптация [поста Akshay Pachaar](https://x.com/akshay_pachaar/status/2035341800739877091) для русскоязычной аудитории.

Полное руководство по CLAUDE.md, пользовательским командам, навыкам, агентам и настройкам разрешений — и как всё это правильно настроить.

---

Большинство пользователей Claude Code относятся к папке `.claude` как к чёрному ящику. Знают, что она существует. Видели, как она появляется в корне проекта. Но никогда не открывали, и тем более не разбирались, что делает каждый файл внутри.

А зря.

Папка `.claude` — это центр управления поведением Claude в вашем проекте. Здесь хранятся ваши инструкции, пользовательские команды, правила доступа и даже память Claude между сессиями. Когда вы понимаете, что где лежит и зачем — вы можете настроить Claude Code именно так, как нужно вашей команде.

Этот гид проходит по всей анатомии папки: от файлов, которые вы будете использовать каждый день, до тех, которые настроите один раз и забудете.

## Две папки, а не одна

Первое, что стоит знать: на самом деле существуют **две** директории `.claude`, а не одна.

Первая живёт внутри вашего проекта, вторая — в домашней директории:

- **Проектная папка** `.claude/` — командная конфигурация. Коммитите в git. Все в команде получают одни и те же правила, команды и политики разрешений.
- **Глобальная папка** `~/.claude/` — ваши личные предпочтения и локальное состояние: история сессий, автоматическая память.

## CLAUDE.md: инструкция для Claude

Это самый важный файл во всей системе. Когда вы запускаете сессию Claude Code, первое, что он читает — `CLAUDE.md`. Файл загружается прямо в системный промпт и остаётся в контексте на протяжении всего разговора.

Проще говоря: **что вы напишете в CLAUDE.md, то Claude и будет делать.**

Если вы скажете Claude всегда писать тесты перед реализацией — он будет. Если напишете «никогда не используй console.log для обработки ошибок, всегда используй кастомный модуль логирования» — он будет это соблюдать каждый раз.

`CLAUDE.md` в корне проекта — самая типичная настройка. Но вы также можете иметь:
- `~/.claude/CLAUDE.md` — глобальные предпочтения для всех проектов
- `CLAUDE.md` внутри поддиректорий — правила для конкретных папок

Claude читает все и объединяет.

### Что писать в CLAUDE.md

Большинство людей пишут либо слишком много, либо слишком мало. Вот что работает.

**Писать:**
- Команды сборки, тестирования и линтинга (`npm run test`, `make build` и т.д.)
- Ключевые архитектурные решения («мы используем monorepo с Turborepo»)
- Неочевидные подводные камни («TypeScript strict mode включён, неиспользуемые переменные — это ошибки»)
- Конвенции импорта, паттерны именования, стили обработки ошибок
- Структуру файлов и папок основных модулей

**Не писать:**
- То, что уже есть в конфиге линтера или форматтера
- Полную документацию, на которую можно дать ссылку
- Длинные абзацы с теорией

**Держите CLAUDE.md до 200 строк.** Файлы длиннее начинают съедать слишком много контекста, и соблюдение инструкций Claude реально падает.

## CLAUDE.local.md для личных настроек

Иногда у вас есть предпочтение, специфичное только для вас, а не для всей команды. Может, вы предпочитаете другой test runner или хотите, чтобы Claude всегда открывал файлы определённым образом.

Создайте `CLAUDE.local.md` в корне проекта. Claude читает его вместе с основным `CLAUDE.md`, и он автоматически добавляется в `.gitignore` — ваши личные настройки никогда не попадут в репозиторий.

## Папка rules/: модульные инструкции, которые масштабируются

`CLAUDE.md` отлично работает для одного проекта. Но когда команда растёт, вы оказываетесь с 300-строчным `CLAUDE.md`, который никто не поддерживает и все игнорируют.

Папка `rules/` решает эту проблему.

Каждый markdown-файл внутри `.claude/rules/` загружается вместе с вашим `CLAUDE.md` автоматически. Вместо одного гигантского файла вы разделяете инструкции по областям:

```
.claude/rules/
├── api-conventions.md
├── testing.md
├── security.md
└── frontend.md
```

Каждый файл остаётся фокусированным и легко обновляемым. Тот, кто отвечает за API-конвенции, редактирует `api-conventions.md`. Тот, кто отвечает за стандарты тестирования — `testing.md`. Никто не наступает друг другу на ноги.

### Правила по пути (path-scoped rules)

Настоящая сила — в правилах, привязанных к путям. Добавьте YAML frontmatter-блок в файл правила, и оно активируется только при работе с подходящими файлами:

```yaml
---
paths:
  - src/api/**
  - src/handlers/**
---
```

Claude не загрузит этот файл, когда редактирует React-компонент. Только когда работает внутри `src/api/` или `src/handlers/`. Правила без поля `paths` загружаются безусловно, каждую сессию.

Это правильный паттерн, когда ваш `CLAUDE.md` начинает казаться переполненным.

## Папка commands/: ваши пользовательские слеш-команды

Из коробки Claude Code имеет встроенные слеш-команды вроде `/help` и `/compact`. Папка `commands/` позволяет добавить свои.

Каждый markdown-файл в `.claude/commands/` становится слеш-командой.

Файл `review.md` создаёт `/project:review`. Файл `fix-issue.md` создаёт `/project:fix-issue`. Имя файла = имя команды.

Пример — создайте `.claude/commands/review.md`:

```markdown
Review the following diff for potential issues:

`!git diff --staged`

Focus on:
- Security vulnerabilities
- Performance problems
- Code style violations
```

Теперь запустите `/project:review` в Claude Code — и он автоматически подставит реальный git diff в промпт. Синтаксис `` `!команда` `` запускает shell-команды и встраивает вывод. Именно это делает эти команды по-настоящему полезными, а не просто сохранённым текстом.

### Передача аргументов

Используйте `$ARGUMENTS` для передачи текста после имени команды:

```markdown
Fix the following GitHub issue:

`!gh issue view $ARGUMENTS`
```

Запуск `/project:fix-issue 234` подтягивает содержимое issue 234 прямо в промпт.

### Личные vs. проектные команды

- **Проектные** в `.claude/commands/` — коммитятся и расшариваются с командой
- **Личные** в `~/.claude/commands/` — доступны везде как `/user:command-name`

Полезные личные команды: помощник для daily standup, генератор коммит-сообщений по вашей конвенции, быстрый security-скан.

## Папка skills/: переиспользуемые рабочие процессы

Команды из `commands/` — это вещи, которые вы запускаете вручную. **Навыки (skills)** — другое. Навыки — это рабочие процессы, которые Claude может запускать **сам**, без вашей слеш-команды, когда задача соответствует описанию навыка.

Каждый навык живёт в своей поддиректории с файлом `SKILL.md`:

```
.claude/skills/
└── security-review/
    ├── SKILL.md
    └── DETAILED_GUIDE.md
```

`SKILL.md` использует YAML frontmatter для описания, когда его применять:

```yaml
---
name: security-review
description: Reviews code for security vulnerabilities
triggers:
  - security review
  - check for vulnerabilities
---
```

Когда вы говорите «проверь этот PR на уязвимости», Claude читает описание, распознаёт соответствие и автоматически вызывает навык. Можно вызвать и явно через `/security-review`.

**Ключевое отличие от команд:** навыки могут содержать сопроводительные файлы. Ссылка `@DETAILED_GUIDE.md` подтягивает подробный документ, лежащий рядом с `SKILL.md`. Команды — это одиночные файлы. Навыки — это пакеты.

Личные навыки хранятся в `~/.claude/skills/` и доступны во всех проектах.

## Папка agents/: специализированные субагенты

Когда задача достаточно сложна, чтобы использовать выделенного специалиста, вы можете определить персону субагента в `.claude/agents/`. Каждый агент — это markdown-файл со своим системным промптом, доступом к инструментам и предпочтением модели:

```yaml
---
name: code-reviewer
model: haiku
tools:
  - Read
  - Grep
  - Glob
---
```

Когда Claude нужно провести code review, он запускает этого агента в отдельном изолированном контекстном окне. Агент делает свою работу, сжимает результаты и отчитывается. Ваша основная сессия не засоряется тысячами токенов промежуточных исследований.

Поле `tools` ограничивает возможности агента. Security-аудитору нужны только `Read`, `Grep` и `Glob`. Ему незачем записывать файлы. Это ограничение намеренное и его стоит прописывать явно.

Поле `model` позволяет использовать более дешёвую и быструю модель для фокусных задач. **Haiku** справляется с большинством read-only задач. Оставьте **Sonnet** и **Opus** для работы, которая действительно в них нуждается.

Личные агенты хранятся в `~/.claude/agents/` и доступны во всех проектах.

## settings.json: разрешения и конфигурация проекта

Файл `settings.json` внутри `.claude/` контролирует, что Claude может и не может делать. Здесь вы определяете, какие инструменты Claude может запускать, какие файлы читать, и нужно ли спрашивать разрешение перед выполнением определённых команд.

```json
{
  "$schema": "https://schema.claude.ai/settings.json",
  "allow": [
    "Bash(npm run *)",
    "Bash(make *)",
    "Bash(git *)",
    "Read", "Write", "Edit", "Glob", "Grep"
  ],
  "deny": [
    "Bash(rm -rf *)",
    "Bash(curl *)",
    "Read(.env*)",
    "Read(secrets/*)"
  ]
}
```

Что делает каждая часть:

- **`$schema`** — включает автодополнение и inline-валидацию в VS Code или Cursor. Всегда включайте.
- **`allow`** — команды, которые выполняются без запроса подтверждения.
- **`deny`** — команды, заблокированные полностью, что бы ни случилось.

Если что-то нет ни в одном списке — Claude спрашивает перед выполнением. Эта «середина» намеренная. Она даёт вам подушку безопасности, не требуя предугадывать каждую возможную команду заранее.

**`settings.local.json`** — то же, что `CLAUDE.local.md`, но для разрешений. Не коммитится.

## Глобальная папка ~/.claude/

Вы не часто взаимодействуете с этой папкой, но полезно знать, что в ней.

- **`~/.claude/CLAUDE.md`** — загружается в каждую сессию Claude Code, во всех проектах. Хорошее место для личных принципов кодирования, предпочитаемого стиля или чего угодно, что вы хотите, чтобы Claude помнил вне зависимости от репозитория.
- **`~/.claude/projects/`** — хранит транскрипты сессий и автоматическую память по проектам. Claude Code автоматически сохраняет заметки для себя: обнаруженные команды, паттерны, архитектурные инсайты. Они сохраняются между сессиями. Просматривайте и редактируйте через `/memory`.
- **`~/.claude/commands/`** и **`~/.claude/skills/`** — личные команды и навыки для всех проектов.

## Практический план настройки

Если начинаете с нуля, вот прогрессия, которая работает:

1. **Шаг 1.** Запустите `/init` в Claude Code. Он сгенерирует стартовый `CLAUDE.md`, прочитав ваш проект. Отредактируйте до самого важного.

2. **Шаг 2.** Добавьте `.claude/settings.json` с правилами allow/deny, подходящими для вашего стека. Минимум — разрешите ваши run-команды и запретите чтение `.env`.

3. **Шаг 3.** Создайте одну-две команды для рабочих процессов, которые вы делаете чаще всего. Code review и исправление issues — хорошие стартовые точки.

4. **Шаг 4.** По мере роста проекта и перегрузки `CLAUDE.md` — начните разделять инструкции на файлы в `.claude/rules/`. Привязывайте к путям, где это имеет смысл.

5. **Шаг 5.** Добавьте `~/.claude/CLAUDE.md` с вашими личными предпочтениями. Это может быть что-то вроде «всегда пиши типы перед реализацией» или «предпочитай функциональные паттерны классовым».

Это реально всё, что нужно для 95% проектов. Навыки и агенты пригодятся, когда у вас есть повторяющиеся сложные рабочие процессы, которые стоит упаковать.

## Ключевой инсайт

Папка `.claude` — это, по сути, протокол для объяснения Claude: кто вы, что делает ваш проект и каким правилам следовать. Чем яснее вы это определите, тем меньше времени тратите на корректировку Claude и тем больше времени он тратит на полезную работу.

**`CLAUDE.md` — ваш файл с максимальным рычагом. Начните с него. Всё остальное — оптимизация.**

Начните с малого, улучшайте по ходу и относитесь к этому как к любой другой части инфраструктуры проекта: то, что приносит дивиденды каждый день, когда правильно настроено.
