# Вайб-кодинг лайфхак 6 — Skills

Тред для Threads. Серия "Вайб-кодинг для бизнеса", лайфхак №6.

Развивает тему лайфхака №3 (Навыки): там я дал базу из 4 правил. Здесь — продвинутый чеклист из 20 правил из официальной документации Anthropic.

Источник: [[Лайфхаки при создании Claude Code skills]]

---

ВАЙБКОДИНГ ЛАЙФХАК №6

Ты уже пользуешься чужими skills в Claude Code. А свои писать не умеешь (зуб даю!).

Я тоже совершал эти ошибки: писал навыки с помощью агента, не вычитывал, собирал в навыке целый домен, а не короткий воркфлоу.

Но тут я собрал всё, что нужно знать о создании своих навыков, чтобы они были не просто балластом в контексте, а реально полезным дополнением.

Сохраняй, разбираем по порядку. И напиши в конце, сколько правил из 20 ты пока не использовал.

---

Правило 1. Навык должен быть коротким.

Не пиши в нём то, что модель и так знает. Что такое PDF. Что такое JSON. Как работает API.

Каждый токен в навык отъедает контекст у диалога и других инструкций. Тут работает то же самое, что и для CLAUDE.md.

---

Правило 2. В описании пиши КОГДА использовать, не только ЧТО.

Claude выбирает навык из набора по описанию. Если в нём нет триггеров, навык не запустится в нужный момент.

Шаблон: "Создаёт КП. Use when user asks to prepare, rewrite, price, or structure a proposal".

И эта часть должна быть внутри yaml-секции, в поле description!

---

Правило 3. Никаких helper, utils, tools.

И documents, files, data — тоже плохо.

Anthropic рекомендует использовать герундий: processing-pdfs, writing-proposals, qualifying-leads.

По имени должно быть понятно, что навык делает и для кого. Если непонятно — переименуй прямо сейчас.

---

Правило 4. Один навык — одна работа.

Не делай content-marketing на все случаи жизни. Лучше три отдельных: creating-content-plan, writing-posts, processing-feedback.

Узкий навык хорошо знает одну работу. Универсальный — не знает ничего.

---

Правило 5. Progressive disclosure.

Если навык требует разного в разных ситуациях — сделай оглавление со ссылками в SKILL.md. Детали — в reference-файлах рядом.

Загружается только то, что нужно для конкретной задачи. Иначе модель утонет в инструкциях, которые ей в этот раз не пригодятся.

---

Правило 6. Цепочки ссылок ломают навык.

SKILL.md → advanced.md → details.md — Claude может не дочитать до конца.

Все важные reference-файлы должны быть видны напрямую из SKILL.md, на один уровень глубины. Дерево плоское, не иерархическое.

---

Правило 7. Воркфлоу — по шагам.

"Подготовь хорошее КП" — модель догадается как умеет.

Лучше так: 1) определи цель клиента, 2) вытащи интеграции, 3) разбей на фазы, 4) пометь допущения, 5) обоснуй цену.

Чем уже шаги — тем меньше разброс на выходе.

---

Правило 8. Степень свободы под риск задачи.

Для написания поста — гибкие правила, пусть модель решает.

Для миграции базы — точная команда и запрет менять флаги.

Чем выше цена ошибки, тем жёстче инструкция. Творчество уместно не везде.

---

Правило 9. Валидационный цикл.

Сгенерировал → проверил по чеклисту → исправил → проверил снова.

Модель пишет хорошо с первого раза в 60% случаев. С чеклистом и циклом — почти всегда.

Особенно для документов, кода, JSON и баз знаний.

---

Правило 10. Шаблон выходного формата.

Если на выходе должно быть КП, отчёт, JSON, таблица — добавь отдельным файлом шаблон.

"Раздел Задача клиента, раздел Решение, раздел Этапы, раздел Цена".

Без шаблона результат разный каждый раз. С шаблоном — стабильный.

---

Правило 11. Конкретные примеры, не пожелания.

Не "сделай красиво и по делу", а пара input → output.

Покажи реальный запрос и реальный идеальный ответ. Модель хорошо копирует паттерны, которые видит. И плохо угадывает паттерны, которых не видит.

---

Правило 12. Не пихай в навык устаревающую инфу.

"До августа использовали v1, после — v2" — через год это превратится в кашу.

Текущий метод — в основной части. История — в отдельном блоке Old patterns. Навык должен жить дольше, чем твоя сегодняшняя голова.

---

Правило 13. Одна терминология.

Не путай endpoint, URL, route, path для одного и того же.

Не смешивай эскалацию, handoff, передачу оператору.

Выбери один термин и используй его везде. Консистентность снижает шанс, что модель спутается.

---

Правило 14. Детерминированные операции — в скрипты.

Если действие всегда одинаковое — пусть его делает Python, а не модель каждый раз заново.

Так быстрее, дешевле и надёжнее. Claude хорош в творчестве. В рутине лучше bash и python с понятным выводом.

---

Правило 15. Скрипты сами обрабатывают ошибки.

Anthropic называет это Solve, don't punt.

Если скрипт упадёт с FileNotFoundError и кинет исключение в модель — она будет гадать. Лучше try/except и понятный return.

Навык должен быть устойчивым к неидеальным входам.

---

Правило 16. Зависимости явно.

В Claude API нет network access и runtime install. На claude.ai возможности другие.

Пиши прямо: требует pypdf. Если недоступен — остановись и отчитайся.

Навык, который молча падает на отсутствующем пакете, хуже, чем навык, который не запустился вовсе.

---

Правило 17. Forward slashes даже на Windows.

scripts/helper.py — работает везде.

scripts\helper.py — ломается в Unix.

Мелочь, но именно из таких мелочей складывается навык, которым можно поделиться без оговорок "у меня работает".

---

Правило 18. Тестируй на реальных сценариях.

Не на happy path. Минимум три кейса:

- типовой запрос,
- запрос с неполными данными,
- запрос с edge case или конфликтом правил.

Если навык работает только на идеальных входах — он не работает.

---

Правило 19. Навык делается итеративно.

Сначала реши задачу руками один раз. Потом второй. Заметь, какой контекст и правила приходится повторять.

Только потом упакуй повторяющееся в навык.

Преждевременный навык = мёртвый навык. Сначала паттерн, потом упаковка.

---

Правило 20. Один default плюс escape hatch.

"Use pypdf, pdfplumber, PyMuPDF or another library" — модель потеряется.

Дай один путь по умолчанию и одно исключение для специфичного случая.

"Use pdfplumber for text. For scanned PDFs use pdf2image with pytesseract". Всё.

---

Чеклист перед публикацией навык:

название конкретное, без helper
description с триггерами
SKILL.md без воды
детали в reference, на один уровень
воркфлоу по шагам
output template
concrete examples
validation loop
скрипты для рутины
тесты на 3+ сценариях

Сам сейчас прохожу свои старые skills по этому списку. Половина не пройдёт.

## Черновые заметки

- Проверить длину каждого поста (Threads лимит 500 символов) — все в пределах ~250–460.
- Пост 1 (HEADER + хук): визуал — скриншот директории `.claude/skills/` со множеством папок.
- Пост 5 (progressive disclosure): уместна схема SKILL.md + references/ + scripts/.
- Пост 15 (Solve, don't punt): можно добавить кодовый карусель-слайд для IG/Slidev параллельно.
- Чеклист (предпоследний): кандидат на отдельную карусель в IG/LinkedIn.
- CTA сегментирует активных пользователей Claude Code, даёт материал для follow-up.
- Параллельно можно добавить версию для Telegram (один длинный пост) — как в лайфхаке №5 Git и №3 Навыки.
- Часть лайфхаков (5, 14, 18) хорошо ложатся в учебный модуль AI Coding Training для Dogovor24.
