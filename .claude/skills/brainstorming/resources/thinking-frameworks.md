# Thinking Frameworks Guide

Фреймворки мышления для brainstorming sessions.

## Бизнес-стратегия

### First Principles Thinking

Разложить проблему на базовые элементы, отбросив допущения.

**Когда использовать:**
- Сложная проблема с множеством assumptions
- Нужно найти нестандартное решение
- "Так всегда делали" не работает

**Как применять:**
1. Что мы точно знаем как факт?
2. Какие assumptions мы делаем?
3. Можем ли мы достичь цели без этих assumptions?

**Пример:**
- Факт: Клиенты нуждаются в автоматизации
- Assumption: Им нужна no-code платформа
- Вопрос: А может им нужен просто консультант + готовые решения?

---

### Jobs To Be Done (JTBD)

Люди не покупают продукты, они "нанимают" их для выполнения "работы".

**Когда использовать:**
- Валидация продуктовой идеи
- Позиционирование на рынке
- Определение конкурентов (неожиданных)

**Как применять:**
1. Какую "работу" пытается сделать клиент?
2. Что он использует сейчас для этой работы?
3. Почему существующее решение его не устраивает?

**Пример (AGIents.pro):**
- Job: "Автоматизировать поддержку клиентов без найма разработчиков"
- Current solution: Консультанты + custom development
- Pain: Долго, дорого, нужен tech expertise

---

### Trend + Need + Technology

Успешная идея = пересечение тренда, реальной потребности и доступной технологии.

**Когда использовать:**
- Оценка timing для запуска продукта
- Валидация startup идеи

**Как применять:**
1. **Trend:** Есть ли растущий тренд? (Google Trends, LinkedIn posts)
2. **Need:** Есть ли реальная боль у аудитории?
3. **Technology:** Технология доступна прямо сейчас?

**Пример (AI automation в 2026):**
- ✅ Trend: AI adoption растёт (Claude, GPT доступны всем)
- ✅ Need: Бизнес хочет автоматизировать, но нет AI expertise
- ✅ Tech: Claude API, LangChain, n8n доступны
- → Good timing для AGIents.pro

---

### 10x vs 10%

Это улучшение на 10% или прорыв в 10 раз?

**Когда использовать:**
- Оценка амбициозности идеи
- Решение "стоит ли это делать"

**10% improvement:**
- Конкурируешь с существующими
- Нужен sales & marketing
- Медленный рост

**10x improvement:**
- Создаёшь новую категорию
- Product-led growth
- Viral potential

**Пример:**
- AI-код-ревью быстрее на 50% = 10% improvement
- AI-код-ревью за $200 вместо $5000 консультанта = 10x на цене

## Технические решения

### Trade-offs Analysis

Каждое решение = набор trade-offs. Вопрос не "что лучше", а "какой trade-off приемлем".

**Когда использовать:**
- Выбор между технологиями
- Архитектурные решения
- Build vs Buy решения

**Как применять:**
1. Что мы получаем?
2. Что мы теряем?
3. Какой trade-off критичен для нас?

**Пример (Go vs Python для backend):**
```
Go:
+ Производительность, concurrency
- Меньше AI/ML библиотек

Python:
+ Богатая экосистема AI/ML
- Медленнее, проблемы с concurrency

Trade-off question: Что важнее - скорость execution или AI capabilities?
Для AGIents → Go для платформы, Python для AI microservice
```

---

### Second-order Effects

Какие неочевидные последствия повлечёт это решение?

**Когда использовать:**
- Архитектурные решения
- Изменения в процессах
- Scaling решения

**Как применять:**
1. Прямое следствие решения
2. Что это изменит ещё?
3. А что это изменит дальше?

**Пример (переход на микросервисы):**
- 1st order: Сервисы независимы
- 2nd order: Нужна service discovery, monitoring
- 3rd order: Нужен DevOps в команде или managed k8s

---

### Build vs Buy vs Partner

Три стратегии получения capability.

**Когда использовать:**
- Нужна новая функциональность
- Выбор технологии

**Матрица решения:**

|             | Build            | Buy               | Partner          |
|-------------|------------------|-------------------|------------------|
| Контроль    | Полный           | Средний           | Минимальный      |
| Время       | Долго            | Быстро            | Средне           |
| Стоимость   | Высокая (dev)    | Средняя (license) | Варьируется      |
| Гибкость    | Максимальная     | Ограничена        | Зависит          |
| Риск        | Tech debt        | Vendor lock-in    | Зависимость      |

**Правило:**
- Build: Core competency, competitive advantage
- Buy: Commodity, не core
- Partner: Нужна экспертиза, которой нет

## Решение проблем

### 5 Whys

Докопаться до root cause через серию "почему?".

**Когда использовать:**
- Debugging сложных проблем
- Поиск корневой причины
- Предотвращение повторения проблемы

**Как применять:**
Спросить "Почему?" 5 раз подряд.

**Пример:**
```
Проблема: Клиенты не активируются после signup

1. Почему? → Не заполняют onboarding
2. Почему? → Onboarding слишком длинный
3. Почему? → Пытаемся собрать все данные сразу
4. Почему? → Боимся потерять пользователя потом
5. Почему? → Нет механизма постепенного onboarding

Root cause: Нужен progressive onboarding, а не all-at-once
```

---

### Inversion

Что если сделать наоборот?

**Когда использовать:**
- Застрял на проблеме
- Все очевидные решения не работают
- Нужна креативность

**Как применять:**
1. Переформулировать задачу наоборот
2. Решить обратную задачу
3. Инвертировать обратно

**Пример:**
```
Задача: Как привлечь больше клиентов?
Inversion: Как отпугнуть всех клиентов?
- Сложный signup
- Непонятное value proposition
- Дорого
- Плохой UX

→ Значит нужно: простой signup, чёткое value prop, доступная цена, хороший UX
```

---

### Constraint Removal

Что если убрать главное ограничение?

**Когда использовать:**
- Чувствуешь, что упираешься в потолок
- Решение кажется невозможным
- Thinking outside the box

**Как применять:**
1. Какое главное ограничение?
2. Что если его нет? Что бы ты сделал?
3. Можем ли мы приблизиться к этому?

**Пример:**
```
Ограничение: Нет времени на разработку
Без ограничения: Нанял бы 10 разработчиков
Приближение: Можно ли купить ready solution? Использовать no-code? Outsource?

Ограничение: Нет денег на маркетинг
Без ограничения: Купил бы рекламу везде
Приближение: Можно ли сделать product-led growth? Viral механику? Partnership?
```

## Применение фреймворков

### Когда НЕ использовать фреймворки

- ❌ Для простых yes/no вопросов
- ❌ Когда ответ очевиден
- ❌ Когда нужна скорость, а не глубина

### Как комбинировать

**Для бизнес-идеи:**
1. JTBD → понять real need
2. Trend + Need + Tech → оценить timing
3. 10x vs 10% → понять амбициозность

**Для технического решения:**
1. Trade-offs → понять что получаем/теряем
2. Second-order Effects → понять последствия
3. Build vs Buy → выбрать стратегию

**Для проблемы:**
1. 5 Whys → найти root cause
2. Constraint Removal → найти creative solution
3. Inversion → проверить альтернативный взгляд

---

## Важно

Фреймворки - это инструменты мышления, а не формулы.

Не нужно:
- ❌ Показывать пользователю "я применяю JTBD"
- ❌ Формально заполнять таблицы
- ❌ Использовать все фреймворки для каждой проблемы

Нужно:
- ✅ Использовать как mental models
- ✅ Показывать результат мышления
- ✅ Выбирать relevant фреймворк для ситуации
