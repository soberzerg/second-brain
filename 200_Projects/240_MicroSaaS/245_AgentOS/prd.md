# PRD: AgentOS — Серверная ОС для AI-агентов

## Обзор приложения

AgentOS — серверная платформа для оркестрации AI-агентов на базе Claude Code. Позволяет вынести работу агентов в облако (VPS), управлять ими через Telegram-ботов и наблюдать за состоянием через веб-дашборд.

Каждый агент привязан к vault (директории проекта на сервере) и доступен через отдельного Telegram-бота. Члены команды получают доступ к нужным агентам через веб-дашборд.

**Ключевая идея:** система — тонкий маршрутизатор, а не умный оркестратор. Вся "интеллектуальная" работа происходит внутри Claude Code, который управляется через CLAUDE.md и настройки конкретного vault.

## Цели и задачи

- Перенести работу AI-агентов с локальной машины на сервер (доступ из любой точки)
- Обеспечить управление агентами через Telegram (оперативная работа) и веб-дашборд (конфигурация)
- Дать возможность команде работать с AI-агентами через индивидуальные Telegram-боты
- Поддерживать диалоговый режим (агент помнит контекст предыдущих сообщений)
- Работать с множеством проектов/vault одновременно

## Целевая аудитория

**Основной сегмент:** Технические фаундеры и тимлиды, которые используют Claude Code для разработки и хотят дать команде доступ к AI-агентам без настройки локального окружения.

**User Persona:**
- Сергей, технический фаундер, ведёт 3-5 проектов одновременно
- Использует Claude Code ежедневно для кодинга, контента, аналитики
- Хочет управлять агентами с телефона через Telegram
- Хочет дать доступ разработчику к агенту Clario, ассистенту — к агенту Second Brain
- Нужна прозрачность: видеть что агенты делают, сколько ресурсов потребляют

## Основные фичи

### MVP (Фаза 1)

1. **Управление агентами (Web)** — CRUD агентов: имя, описание, путь к vault, Telegram bot token. Запуск/остановка.

2. **Telegram-интерфейс** — Получение сообщений через Telegram-бота, маршрутизация в нужный vault, запуск Claude Code CLI (`claude -p --directory --resume`), возврат ответа в Telegram.

3. **Сессии с памятью** — Маппинг `telegram_user_id → claude_session_id`. Диалоговый режим через `--resume`. Команда `/new` для сброса сессии.

4. **Управление доступом (Web)** — Пользователи с ролями (admin, member). Admin создаёт агентов и выдаёт доступ. Member видит и использует только назначенные ему агенты.

5. **Логи и история (Web)** — Лента сообщений: кто, когда, какому агенту, что отправил, что получил. Фильтрация по агенту/пользователю/дате.

6. **Статус агентов (Web)** — Онлайн/оффлайн, текущая активная сессия, последнее сообщение, количество сообщений за день.

### Фаза 2 (после валидации)

1. **Очередь задач** — NATS-очередь для асинхронных задач. Агент получает задачу, выполняет в фоне, уведомляет по завершении в Telegram.

2. **Запланированные задачи (Cron)** — Расписание через веб-дашборд: "каждый день в 9:00 запусти дайджест", "каждый понедельник — weekly synthesis".

3. **Webhooks** — Входящие webhooks от GitHub/GitLab (push, PR, issue) → автоматический запуск агента в контексте проекта.

4. **Мониторинг ресурсов** — Потребление CPU/RAM по агентам, количество токенов Claude Code, стоимость за период.

5. **Файловый менеджер (Web)** — Просмотр файлов vault из дашборда. Read-only для начала, редактирование CLAUDE.md.

### Будущее развитие

- Multi-agent War Room (совет агентов, как в ClaudeClaw)
- Голосовой интерфейс (STT/TTS через PipeCat)
- Hive Mind — общая память между агентами разных vault
- WhatsApp / Slack каналы (помимо Telegram)
- Маркетплейс шаблонов vault (преднастроенные CLAUDE.md + skills)
- Meeting Bot (подключение к Google Meet/Zoom)

## Технический стек

| Компонент | Решение | Обоснование |
|-----------|---------|-------------|
| Backend | Go (Kratos) | Знакомый стек, production-ready, микросервисная архитектура |
| ORM | Ent | Типизированная кодогенерация, миграции, хуки |
| БД | PostgreSQL | Основное хранилище: агенты, пользователи, сессии, логи |
| Кэш | Redis | Сессии, статусы агентов, rate limiting |
| Messaging | NATS | Асинхронные задачи, pub/sub между сервисами (Фаза 2) |
| AI Runtime | Claude Code CLI (subprocess) | `claude -p --directory --resume` через `exec.Command` |
| Frontend | React + Vite + Tailwind + Shadcn | Современный стек, быстрая разработка UI |
| Telegram | Go Telegram Bot API | Нативная Go-библиотека, без Python-зависимостей |
| Хостинг | VPS | Полный контроль, один сервер для MVP |
| Reverse Proxy | Caddy / Nginx | HTTPS для дашборда, проксирование API |

## Концептуальная модель данных

### Основные сущности

- **User** — пользователь системы: `id`, `name`, `email`, `role` (admin/member), `telegram_id`, `created_at`
- **Agent** — AI-агент: `id`, `name`, `description`, `vault_path`, `telegram_bot_token`, `status` (active/inactive), `created_at`
- **AgentAccess** — связь пользователь-агент: `id`, `user_id`, `agent_id`, `granted_at`, `granted_by`
- **Session** — активная Claude Code сессия: `id`, `user_id`, `agent_id`, `claude_session_id`, `started_at`, `last_message_at`
- **MessageLog** — история сообщений: `id`, `session_id`, `user_id`, `agent_id`, `direction` (in/out), `content`, `created_at`, `duration_ms`, `tokens_used`

### Ключевые связи

- User → AgentAccess → Agent: многие-ко-многим (пользователь имеет доступ к нескольким агентам)
- User + Agent → Session: одна активная сессия на пару пользователь-агент
- Session → MessageLog: одна сессия содержит много сообщений

### ER-диаграмма

```
User 1──* AgentAccess *──1 Agent
 │                         │
 └──────* Session *────────┘
            │
            └──* MessageLog
```

## UI/UX Принципы

### Web-дашборд

- **Desktop-first** — рабочий инструмент, не мобильное приложение
- **Минимализм** — таблицы, формы, логи. Никаких декоративных элементов
- **Real-time статусы** — WebSocket для обновления статусов агентов и новых логов
- **Референсы:** Vercel Dashboard (чистота), Coolify (self-hosted управление), Portainer (контейнеры)

### Telegram

- **Нулевой onboarding** — написал боту, получил ответ (если есть доступ)
- **Команды:** `/new` — новая сессия, `/status` — статус агента, `/help` — справка
- **Длинные ответы** — разбивать на несколько сообщений (лимит Telegram: 4096 символов)
- **Markdown** — форматировать ответы Claude в Telegram Markdown

## Аутентификация и безопасность

### Web-дашборд

- **Метод:** JWT-токены (access + refresh)
- **Первый пользователь** — автоматически admin (setup wizard)
- **Приглашения** — admin генерирует invite-link для новых участников

### Telegram

- **Идентификация:** по `telegram_user_id` (привязан к User в системе)
- **Привязка:** пользователь вводит код из дашборда в Telegram-бот для линковки аккаунта
- **Проверка доступа:** при каждом сообщении проверяется `AgentAccess` для пары user+agent

### Защита данных

- **Vault isolation:** агент работает только в своей директории (`--directory`)
- **Secrets:** Telegram bot tokens и Claude API key хранятся в env-переменных, не в БД
- **Rate limiting:** Redis-based, ограничение сообщений на пользователя (предотвращение злоупотреблений)
- **Audit log:** все действия логируются в MessageLog

## Потенциальные вызовы и решения

| Вызов | Влияние | Решение |
|-------|---------|---------|
| Claude Code CLI subprocess может зависнуть | Заблокированный агент, нет ответа пользователю | Timeout на subprocess (5 мин), kill + уведомление пользователя. Healthcheck сессий |
| Длинные ответы Claude (>4096 символов) | Telegram обрезает сообщения | Разбивать на чанки по 4096, отправлять последовательно с задержкой |
| `--resume` сессия может протухнуть | Потеря контекста, ошибка при resume | Fallback: если resume failed, создать новую сессию, уведомить пользователя |
| Параллельные сообщения одному агенту | Race condition, конфликт subprocess | Очередь сообщений на агента (Redis queue), последовательная обработка |
| Claude Code подписка — лимиты | Превышение лимитов при активном использовании | Rate limiting на уровне системы, приоритизация сообщений, мониторинг использования |
| Безопасность vault — агент может выйти за пределы | Доступ к чужим файлам | `--directory` ограничивает scope. В будущем — chroot/Docker для жёсткой изоляции |
| Обновления Claude Code CLI | Может сломать интеграцию | Зафиксировать версию, тестировать обновления перед деплоем |
