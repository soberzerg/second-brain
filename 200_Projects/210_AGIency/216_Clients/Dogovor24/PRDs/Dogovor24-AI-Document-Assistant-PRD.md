# PRD: AI-ассистент по созданию юридических документов Dogovor24

> Коммерческое предложение AGIency для Dogovor24 — интеллектуальная система создания юридических документов через чат-интерфейс с опорой на графовую базу шаблонов и клауз

## 1. Обзор проекта

### Проблема

Dogovor24 предоставляет конструктор документов — сервис с шаблонами договоров, вариантами клауз и условиями их применения. Текущий конструктор работает как пошаговая форма: пользователь выбирает тип документа, отвечает на вопросы, система подставляет нужные пункты. Это создаёт ограничения:

- **Высокий порог входа:** пользователь должен знать, какой тип договора ему нужен, и разбираться в юридической терминологии
- **Линейный процесс:** нет возможности описать ситуацию своими словами и получить подходящий документ
- **Нет проверки рисков:** конструктор собирает документ, но не предупреждает о юридических рисках выбранных формулировок
- **Изолированность от консультаций:** пользователь, получивший юридическую консультацию через amicus, не может сразу перейти к созданию документа из контекста этой консультации
- **Разрозненные знания:** связи между шаблонами, клаузами, условиями их применения и рисками хранятся имплицитно в логике конструктора, а не в явном виде

### Решение

Отдельный микросервис — AI-ассистент по созданию документов с чат-интерфейсом по типу ChatGPT, который:

- Принимает описание ситуации пользователя на естественном языке
- Определяет нужный тип документа и подбирает шаблон из графовой БД
- Задаёт уточняющие вопросы для выбора правильных клауз
- Собирает документ из подходящих клауз с учётом ситуации
- Проверяет итоговый документ на юридические риски (через интеграцию с AI Legal Gateway)
- Отдаёт готовый документ на скачивание или отправку в ЭДО dogovor24 для подписания ЭЦП

> **Принцип:** Сервис отвечает за чат-интерфейс, логику сборки документов из графа и генерацию файлов. Проверка рисков делегируется AI Legal Gateway. ЭДО-подписание — существующей инфраструктуре D24.

### Целевой результат

Сократить время создания юридического документа с **часов (ручной поиск шаблона + заполнение) до минут** (описание ситуации в чате). Повысить качество документов за счёт AI-проверки рисков. Увеличить конверсию пользователей конструктора за счёт естественного интерфейса.

## 2. Контекст: существующая инфраструктура

### Конструктор документов Dogovor24

| Параметр | Значение |
| -------- | -------- |
| Тип | Существующий сервис в инфраструктуре D24 |
| Данные | Шаблоны документов + клаузы (пункты/абзацы) + условия применения клауз |
| Гранулярность | Клауза = отдельный пункт/абзац договора (напр. «п. 3.2 — ответственность за просрочку») |
| Вариативность | Для каждой позиции в шаблоне — несколько вариантов клауз в зависимости от ситуации |

### AI Legal Gateway (юридический ассистент)

| Параметр | Значение |
| -------- | -------- |
| Тип | Отдельный микросервис (Go Kratos + Ent) |
| Зона ответственности | Ответы на юридические вопросы + AI-оценка ответов |
| Роль в документ-ассистенте | Проверка сгенерированных документов на юридические риски |
| Связь | REST/gRPC API |

### ЭДО Dogovor24

| Параметр | Значение |
| -------- | -------- |
| Тип | Основной продукт D24 |
| Функция | Электронное подписание документов ЭЦП |
| Роль в документ-ассистенте | Приём готового документа для подписания контрагентами |

### Наш сервис — AI Document Assistant

| Параметр | Значение |
| -------- | -------- |
| Тип | **Отдельный микросервис**, не часть инфраструктуры D24 |
| Стек | Go (Kratos framework + Ent ORM) |
| Зона ответственности | Чат-интерфейс + графовая БД клауз + сборка документов + генерация файлов |
| Связь с D24 | REST/gRPC API (конструктор, ЭДО) |
| Связь с AI Legal Gateway | REST/gRPC API (проверка рисков) |

### 3.2 Компоненты системы

| Компонент | Тип | Описание |
|-----------|-----|----------|
| **Chat UI** | Frontend (React/Next.js) | Чат-интерфейс по типу ChatGPT с превью документа |
| **Dialogue Agent** | AI-агент | Понимание ситуации, уточняющие вопросы, управление flow |
| **Graph Query Engine** | Сервис | Навигация по графовой БД: поиск шаблонов, подбор клауз, определение рисков |
| **Document Composer** | AI-агент | Сборка документа из клауз, заполнение переменных, генерация файлов |
| **Graph DB (клаузы)** | Графовая БД | Шаблоны → разделы → клаузы → условия → риски |
| **AI Legal Gateway** | Внешний сервис | Проверка сгенерированного документа на юридические риски |

### 3.3 Dialogue Agent — логика диалога

Dialogue Agent ведёт пользователя через процесс создания документа:

**Фаза 1: Понимание ситуации**

```
Пользователь: "Хочу сдать квартиру в аренду, нужен договор"

Dialogue Agent:
  1. Определяет тип документа → Договор аренды жилого помещения
  2. Ищет шаблон в графовой БД
  3. Определяет обязательные параметры, которые нужно уточнить
```

**Фаза 2: Уточняющие вопросы**

```
Ассистент: "Отлично, подготовлю договор аренды. Уточню несколько моментов:

1. Квартира в каком городе?
2. На какой срок аренды? (до 1 года / от 1 года)
3. Будет ли залоговый депозит?
4. Арендатор — физическое или юридическое лицо?"
```

**Фаза 3: Подбор клауз**

```
На основе ответов Graph Query Engine:
  - Выбирает клаузу по сроку: "краткосрочная аренда (до 1 года)"
  - Добавляет клаузу о залоге: "залоговый депозит в размере N"
  - Подбирает клаузу по типу арендатора: "физ. лицо"
  - Определяет региональные особенности: "Алматы"
```

**Фаза 4: Сборка и проверка**

```
Document Composer:
  1. Собирает документ из выбранных клауз
  2. Заполняет переменные (даты, имена, суммы)
  3. Отправляет в AI Legal Gateway на проверку рисков
  4. Показывает пользователю превью + предупреждения о рисках
  5. Пользователь: скачать .docx/.pdf или отправить в ЭДО
```

### 3.4 Навигация по графу — ключевые запросы

| Запрос | Что возвращает | Когда используется |
|--------|---------------|-------------------|
| `template_by_situation(text)` | Подходящий шаблон + score | Фаза 1: определение типа документа |
| `required_params(template_id)` | Список обязательных параметров | Фаза 2: формирование уточняющих вопросов |
| `clauses_for_context(template_id, params)` | Набор клауз с вариациями | Фаза 3: подбор клауз |
| `risks_for_clause(clause_id, context)` | Риски и рекомендации | Фаза 4: предупреждения пользователю |
| `related_templates(template_id)` | Связанные документы | Допродажа: "Вам также может понадобиться..." |

## 4. Графовая БД — модель знаний

### 4.1 Структура графа

```
                    ┌─────────────┐
                    │  Категория   │
                    │  (Аренда,    │
                    │   Трудовой,  │
                    │   Купля-     │
                    │   продажа)   │
                    └──────┬──────┘
                           │ HAS_TEMPLATE
                           ▼
                    ┌─────────────┐
                    │   Шаблон     │◄──── REQUIRES_PARAM ───► ┌──────────┐
                    │  документа   │                          │ Параметр │
                    └──────┬──────┘                          │ (город,  │
                           │ HAS_SECTION                     │  срок,   │
                           ▼                                 │  тип     │
                    ┌─────────────┐                          │  лица)   │
                    │   Раздел     │                          └──────────┘
                    │  (Предмет,   │
                    │   Оплата,    │
                    │   Ответст-   │
                    │   венность)  │
                    └──────┬──────┘
                           │ HAS_CLAUSE_SLOT
                           ▼
                    ┌─────────────┐         ┌─────────────┐
                    │ Clause Slot  │────────▶│  Условие    │
                    │ (позиция     │ WHEN    │ (если срок  │
                    │  для клаузы) │         │  > 1 года,  │
                    └──────┬──────┘         │  если юрлицо│
                           │ HAS_VARIANT    │  ...)       │
                           ▼                └─────────────┘
                    ┌─────────────┐
                    │   Клауза     │────────▶ ┌────────────┐
                    │  (конкретный │ HAS_RISK │   Риск     │
                    │   пункт      │         │ (описание, │
                    │   договора)  │         │  severity, │
                    └─────────────┘         │  mitigation│
                           │                └────────────┘
                           │ REFERENCES_NPA
                           ▼
                    ┌─────────────┐
                    │     НПА      │
                    │  (статья     │
                    │   закона)    │
                    └─────────────┘
```

### 4.2 Типы узлов (Nodes)

| Узел | Описание | Ключевые свойства |
|------|----------|-------------------|
| **Category** | Категория документов | `name`, `description`, `icon` |
| **Template** | Шаблон документа | `name`, `description`, `document_type`, `jurisdiction`, `version`, `is_active` |
| **Section** | Раздел документа | `name`, `order`, `is_required` |
| **ClauseSlot** | Позиция для клаузы в разделе | `order`, `is_required`, `default_variant` |
| **Clause** | Конкретный пункт/абзац договора | `text`, `version`, `language`, `variables[]` |
| **Condition** | Условие применения клаузы | `type` (param_value / combination / custom), `expression` |
| **Risk** | Юридический риск клаузы | `description`, `severity` (low/medium/high/critical), `mitigation`, `affected_party` |
| **Param** | Параметр, влияющий на выбор клауз | `name`, `type` (string/number/enum/boolean), `required`, `options[]` |
| **NPA** | Ссылка на нормативно-правовой акт | `title`, `article`, `paragraph`, `url` |

### 4.3 Типы связей (Edges)

| Связь | От → К | Описание |
|-------|--------|----------|
| `HAS_TEMPLATE` | Category → Template | Категория содержит шаблон |
| `HAS_SECTION` | Template → Section | Шаблон состоит из разделов |
| `HAS_CLAUSE_SLOT` | Section → ClauseSlot | Раздел содержит слоты для клауз |
| `HAS_VARIANT` | ClauseSlot → Clause | Слот имеет варианты клауз |
| `WHEN` | ClauseSlot → Condition | Условие выбора варианта клаузы |
| `HAS_RISK` | Clause → Risk | Клауза имеет потенциальный риск |
| `REFERENCES_NPA` | Clause → NPA | Клауза ссылается на НПА |
| `REQUIRES_PARAM` | Template → Param | Шаблон требует параметр для настройки |
| `RELATED_TO` | Template → Template | Связанные шаблоны документов |
| `CONFLICTS_WITH` | Clause → Clause | Взаимоисключающие клаузы |
| `DEPENDS_ON` | Clause → Clause | Клауза зависит от другой |

### 4.4 Пример: Договор аренды квартиры

```
Category: "Аренда"
  └── Template: "Договор аренды жилого помещения"
        ├── REQUIRES_PARAM → Param: "rental_term" (enum: short/long)
        ├── REQUIRES_PARAM → Param: "city" (string)
        ├── REQUIRES_PARAM → Param: "tenant_type" (enum: individual/legal)
        ├── REQUIRES_PARAM → Param: "deposit" (boolean)
        │
        ├── Section: "1. Предмет договора" (order: 1)
        │     └── ClauseSlot: "предмет_аренды" (required: true)
        │           ├── Clause: "стандартный_предмет"
        │           └── Clause: "предмет_с_имуществом" (если сдаётся с мебелью)
        │
        ├── Section: "3. Арендная плата" (order: 3)
        │     ├── ClauseSlot: "размер_оплаты" (required: true)
        │     │     ├── Clause: "фикс_оплата" — WHEN: любой
        │     │     └── Clause: "оплата_с_индексацией" — WHEN: rental_term = "long"
        │     │           └── Risk: "Индексация может привести к спорам"
        │     │                 severity: medium
        │     │                 mitigation: "Зафиксировать максимальный %"
        │     │
        │     └── ClauseSlot: "залоговый_депозит" (required: false)
        │           └── WHEN: deposit = true
        │           └── Clause: "залог_стандартный"
        │                 └── REFERENCES_NPA → "ГК РК, ст. 544"
        │
        └── Section: "7. Ответственность сторон" (order: 7)
              └── ClauseSlot: "штрафные_санкции"
                    ├── Clause: "мягкие_штрафы"
                    │     └── Risk: "Слабая защита арендодателя"
                    │           severity: low, affected_party: "landlord"
                    └── Clause: "жёсткие_штрафы"
                          └── Risk: "Может быть оспорено по ст. 353 ГК РК"
                                severity: high, affected_party: "tenant"
```

### 4.5 Импорт данных из конструктора D24

**Источник:** существующий конструктор документов Dogovor24

**Процесс миграции:**

1. Получить дамп шаблонов конструктора (API или экспорт БД)
2. Парсинг структуры: шаблоны → разделы → клаузы → условия
3. Обогащение AI: автоматическая разметка рисков и ссылок на НПА
4. Загрузка в графовую БД
5. Ручная верификация юристом (выборочная проверка)

**ETL pipeline:**

```
Конструктор D24 (SQL)
       │
       ▼ Export API / DB dump
  ┌──────────────────┐
  │  ETL Transform    │
  │  - Нормализация   │
  │  - Дедупликация   │
  │  - AI-обогащение  │
  └────────┬─────────┘
           │
           ▼
  ┌──────────────────┐
  │   Graph DB        │
  │   (Neo4j /        │
  │    Apache AGE)    │
  └──────────────────┘
```

### 4.6 Технические требования к графовой БД

| Параметр | Требование |
|----------|-----------|

| Язык запросов | Cypher (Neo4j) или openCypher (AGE) |

| Версионирование | Хранение версий клауз (при обновлении НПА) |
| Кэширование | Часто используемые подграфы в Redis |

## 5. Интеграция с AI Legal Gateway

### 5.1 Проверка рисков документа

После сборки документа Document Composer отправляет его в AI Legal Gateway для проверки:

**POST `/api/ai-legal/documents/risk-check`**

```json
{
  "document_id": "uuid",
  "template_id": "uuid",
  "clauses": [
    {
      "clause_id": "uuid",
      "slot": "штрафные_санкции",
      "text": "В случае просрочки оплаты арендатор уплачивает..."
    }
  ],
  "context": {
    "document_type": "rental_agreement",
    "parties": ["individual", "individual"],
    "jurisdiction": "almaty"
  }
}
```

**Response:**

```json
{
  "document_id": "uuid",
  "overall_risk": "medium",
  "risks": [
    {
      "clause_id": "uuid",
      "severity": "high",
      "description": "Штрафные санкции в п. 7.2 могут быть признаны чрезмерными",
      "legal_reference": "ГК РК, ст. 353",
      "recommendation": "Снизить размер пени до 0.1% в день или использовать мягкий вариант клаузы",
      "alternative_clause_id": "uuid"
    }
  ],
  "warnings": [
    {
      "type": "missing_clause",
      "description": "Рекомендуется добавить пункт о досудебном урегулировании споров"
    }
  ]
}
```

### 5.2 Связь с контекстом консультации

Если пользователь ранее получил консультацию через AI Legal Gateway, контекст этой консультации может использоваться для предзаполнения параметров документа:

```
Пользователь получил консультацию:
"Как правильно сдать квартиру в аренду?"
       │
       ▼ контекст (тип: аренда, регион: Алматы, тип лица: физ.)
       │
       ▼
```

## 9. Функциональные требования

### 9.1 Must Have (P0)

- Как **пользователь**, я хочу описать ситуацию в чате на русском языке и получить подходящий шаблон документа, чтобы не разбираться в типах договоров самостоятельно
- Как **пользователь**, я хочу отвечать на уточняющие вопросы ассистента, чтобы документ был настроен под мою ситуацию
- Как **пользователь**, я хочу видеть превью документа в реальном времени по мере заполнения, чтобы понимать, что получится
- Как **пользователь**, я хочу видеть предупреждения о юридических рисках в документе, чтобы принять осознанное решение
- Как **пользователь**, я хочу скачать готовый документ в формате .docx или .pdf
- Как **пользователь**, я хочу отправить готовый документ в ЭДО dogovor24 для подписания ЭЦП
- Как **система**, Dialogue Agent корректно определяет тип документа по описанию ситуации с точностью >= 85%
- Как **система**, Graph Query Engine подбирает клаузы, соответствующие параметрам пользователя

### 9.2 Should Have (P1)

- Как **пользователь**, я хочу переключать варианты клауз вручную (например, выбрать «мягкие штрафы» вместо «жёстких»)
- Как **пользователь**, я хочу задавать вопросы в чате на казахском языке
- Как **пользователь**, я хочу получить документ из контекста предыдущей юридической консультации, чтобы не повторять информацию
- Как **пользователь**, я хочу редактировать текст отдельных пунктов документа прямо в интерфейсе
- Как **пользователь**, я хочу видеть, на какие статьи законов опирается каждый пункт документа
- Как **администратор**, я хочу добавлять новые шаблоны и клаузы в графовую БД без участия разработчиков

### 9.3 Nice to Have (P2)

- Как **пользователь**, я хочу загрузить существующий договор (скан/фото) для анализа и сравнения с шаблоном
- Как **пользователь**, я хочу получить рекомендации связанных документов («Вам также может понадобиться доверенность на подписание»)
- Как **система**, AI предлагает релевантные платные услуги на dogovor24.kz в конце процесса
- Как **администратор**, я хочу видеть аналитику по популярным шаблонам и типичным выборам клауз
- Как **система**, кэширование частых комбинаций параметров для мгновенной сборки типовых документов

## 11. Метрики успеха

### 11.1 Ключевые метрики

| Метрика | Определение | Целевое значение |
|---------|------------|------------------|
| **Template Match Accuracy** | % правильно определённых типов документов | > 85% (Phase 1), > 95% (Phase 3) |
| **Document Completion Rate** | % начатых документов, доведённых до скачивания/ЭДО | > 60% |
| **Time to Document** | Время от первого сообщения до готового документа | < 5 мин (типовой документ) |
| **Risk Detection Rate** | % документов, где риски были выявлены и показаны | > 90% (от документов с рисками) |
| **User Satisfaction** | Оценка после создания документа (1-5) | > 4.0 |
| **ЭДО Conversion** | % документов, отправленных в ЭДО (vs скачивание) | > 30% |

### 11.2 Операционные метрики

| Метрика | Описание |
|---------|----------|
| Средняя длина диалога | Количество сообщений до готового документа |
| Drop-off по шагам | На каком вопросе пользователи уходят |
| Популярные шаблоны | Топ-10 запрашиваемых типов документов |
| Clause override rate | Как часто пользователи меняют предложенные клаузы |
| Graph query latency p95 | 95-й перцентиль времени запросов к графовой БД |

## 12. Non-Goals (вне скопа)

Система **НЕ будет**:

- Заменять конструктор документов D24 (работает параллельно как альтернативный интерфейс)
- Генерировать документы с нуля без шаблонов (только из графовой базы клауз)
- Проводить юридическую экспертизу (только автоматическая проверка рисков на основе графа и AI Legal Gateway)
- Хранить подписанные документы (это функция ЭДО D24)
- Работать без авторизации (требуется аккаунт D24)
- Поддерживать юрисдикции кроме РК
- Обрабатывать документы не из базы шаблонов (загрузка произвольных документов — P2)

## 13. Риски и митигация

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| **Неправильный подбор клауз → юридически некорректный документ** | Средняя | Критическое | Обязательная проверка рисков через AI Legal Gateway; дисклеймер; рекомендация проверки юристом |
| **Низкое качество графовой БД после импорта** | Средняя | Высокое | Этапный импорт (1 категория за раз); ручная верификация юристом; A/B-тестирование |
| **Сложность структуры конструктора D24 для маппинга в граф** | Высокая | Среднее | Ранний доступ к данным конструктора; прототип маппинга до начала основной разработки |
| **Пользователь описывает ситуацию слишком абстрактно** | Высокая | Среднее | Продуманные уточняющие вопросы; fallback на каталог шаблонов; примеры запросов |
| **Высокий drop-off в диалоге** | Средняя | Среднее | UX-тестирование; минимизация вопросов; предзаполнение из контекста |
| **Рассинхрон графовой БД и конструктора D24** | Средняя | Высокое | Автоматическая синхронизация (webhook/cron); алерты при расхождении |
| **Регуляторные риски** | Низкая | Высокое | Дисклеймер «документ подготовлен автоматически»; рекомендация проверки юристом |

## 14. Технический стек

### 14.1 Backend

| Компонент | Технология | Обоснование |
|-----------|-----------|-------------|

| API-транспорт | gRPC + REST (Kratos HTTP/gRPC dual) | Type-safe контракты |

| LLM | Claude Sonnet 4.6 / GPT-4o | Понимание ситуации, генерация вопросов |
| LLM (simple) | Claude Haiku / GPT-4o-mini | Заполнение переменных, простые запросы |
| Генерация файлов | go-docx / wkhtmltopdf | Сборка .docx и .pdf из клауз |
| Real-time | WebSocket (Kratos WebSocket) | Стриминг ответов в чат |
| Кэширование | Redis | Кэш популярных подграфов, сессии |
| Хранение файлов | S3-compatible (MinIO / Yandex Object Storage) | Сгенерированные документы |

### 14.2 Frontend

| Компонент | Технология | Обоснование |
|-----------|-----------|-------------|

| UI-кит | Tailwind CSS + shadcn/ui | Консистентный дизайн |
| Чат | WebSocket + custom components | Real-time |

| i18n | i18next | Русский / Казахский |

### 14.3 Deployment

- Отдельный Docker-контейнер (backend) + Docker-контейнер (frontend)
- Связь с D24 только через API
- CI/CD: собственный pipeline (GitHub Actions / GitLab CI)
- Графовая БД: отдельный контейнер Neo4j (или AGE внутри PostgreSQL)

## 15. Этапы внедрения

### Phase 1: MVP — одна категория документов (8-10 недель)

| Неделя | Задача |
|--------|--------|
| 1-2 | Проектирование архитектуры Go Kratos сервиса, модель графа, proto-контракты |
| 3-4 | Импорт данных из конструктора D24: **одна категория** (рекомендация: Договоры аренды) → графовая БД |
| 5-6 | Dialogue Agent + Graph Query Engine + Document Composer |
| 7-8 | Чат-интерфейс (React) + split-view с превью документа |
| 9 | Интеграция: AI Legal Gateway (проверка рисков) + ЭДО D24 (подписание) |
| 10 | Тестирование на реальных сценариях, калибровка |

**Объём Phase 1:**

- **1 категория документов** (на выбор заказчика — рекомендация: Договоры аренды как высокочастотная категория)
- Только русский язык
- Скачивание .docx + отправка в ЭДО
- Проверка рисков через AI Legal Gateway

**Критерии успеха Phase 1:**

- Template Match Accuracy > 80% для выбранной категории
- Document Completion Rate > 50%
- Time to Document < 7 мин
- Нет критических ошибок в сгенерированных документах

### Phase 2: Расширение категорий (4-6 недель на каждую)

- Поэтапное добавление категорий документов **по одной** (порядок — по согласованию с заказчиком)
- Каждая новая категория: импорт шаблонов → граф → Dialogue Agent prompts → калибровка
- Интеграция с контекстом юридических консультаций
- Казахский язык (P1)
- Редактирование клауз в интерфейсе
- Административная панель для управления графом

**Критерии перехода на Phase 2:**

- Phase 1 работает 4+ недели
- Document Completion Rate стабильно > 50%
- 100+ созданных документов по первой категории

### Phase 3: Масштабирование (ongoing)

- Все категории конструктора D24 в графовой БД
- Загрузка и анализ существующих документов (P2)
- Рекомендации связанных документов
- Кэширование типовых комбинаций
- AI-автообновление графа при изменении НПА
- Аналитика: популярные шаблоны, типичные выборы клауз, воронка
